### If one operation causally depends on another, all nodes will see them in the same order. But operations that are independent (concurrent) can be seen in different orders.
### Think in terms of cause → effect:
- If A happens before B, and B depends on A
- everyone must see A before B
- If A and B are unrelated
- different users may see:
- A then B
- or B then A
- That’s allowed

### Real-Life Example
- Chat system (like WhatsApp)
- User A sends:
- "Hey"
- Then sends:
- "Are you there?"
- 👉 Message 2 depends on Message 1
- ✔ Causal consistency ensures:
- No one sees:
- "Are you there?" before "Hey"

### But:
- User B sends:
- "Hello everyone"
- This is independent
- So users might see:
- A's messages first, then B's
- OR B's message first
- ✔ Both are valid

### What is “Causality”? Two operations are causally related if:
- Program Order
    - Same client:
    - write(x=1) → write(x=2)
    - 👉 Second depends on first
- Read-After-Write
    - User A writes post
    - User B reads it
    - User B comments on it
    - 👉 Comment depends on post
- Transitive Dependency
    - A → B → C
    - 👉 Then A → C

### How Systems Implement It, To track causality, systems use:
- Vector Clocks (Most Important)
    - Each node maintains a vector like:
    - [A:1, B:2, C:0]
    - Tracks which events a node has seen
    - Helps detect:
      - causal dependency
      - concurrent updates
- Dependency Tracking
    - When writing:
      - Attach metadata (vector clock / version)
    - When reading:
      - Ensure all dependencies are satisfied before returning data
- Delayed Visibility
  - If a node receives:
  - Operation B (depends on A)
  - but hasn’t seen A yet
  - 👉 It waits until A arrives
