### We have a distributed Wallet Service, Multiple replicas:
- Node A (primary region)
- Node B (secondary region / offline-capable)

- example stored record:
- The vector clock is NOT a separate key, It is metadata attached to a data record
```json
{
  "userId": 123,
  "balance": 100,
  "vc": {
    "A": 0,
    "B": 0
  }
}
```

- example nodeA and nodeB assume A -> US-East Server, B-> IN-North Server 
### Flow 1: Normal Update (No Conflict)
#### Step 1: User adds ₹50 via Node A
- At Node A:
- Apply Rule 1 (local event)
- VC[A] += 1 → [1,0]
- Update:
```
{
  balance: 150,
  vc: [1,0]
}
```
#### Step 2: Replication to Node B
- Node B receives:
- Merge:
- max([0,0], [1,0]) = [1,0]
- Increment (receive event): [1,1]
- Now B stores:
```
{
  balance: 150,
  vc: [1,1]
}
```
- ✅ Result Both nodes are consistent:
- No conflict
- Proper ordering maintained

### Flow 2: Concurrent Updates (REAL PROBLEM)
- Now things get interesting.
- Initial State
- balance: 100
- vc: [0,0]
#### Step 1: User updates via Node A
- Add ₹50:
- At A:
- VC → [1,0]
- balance = 150

#### Step 2: At SAME TIME, Node B processes another request
- Add ₹30:
- At B:
- VC → [0,1]
- balance = 130
- 🚨 Now we have TWO versions
- Node	Balance	VC
- A	150	[1,0]
- B	130	[0,1]

#### Step 3: Replication happens
- A receives B’s version
- B receives A’s version

#### 🧠 Conflict Detection Using Vector Clock
- Compare:
- [1,0] vs [0,1]
- 👉 Neither dominates
- ✔ This means:
- Concurrent updates (conflict)

#### ❗ Why This is Critical
- If you didn’t detect this:
- 👉 You might overwrite:
- ₹150 with ₹130 ❌
- or vice versa ❌
- 👉 Money lost → unacceptable

#### Important Reality Check, 👉 In real fintech systems, you usually DON’T use vector clocks, Instead you use:
- transactions
- locking
- ledger-based systems
- Why?, 👉 Because:
- “Money cannot tolerate conflicts”

#### Where Vector Clocks DO Fit
- Offline-first apps
- Eventually consistent systems
- Shopping carts
- Social feeds

### Types of application
- collaborative editing, social media feeds, and chat apps

### Applications Using Causal Consistency:
#### Collaborative Editing Tools: Tools like Google Docs use it to ensure that if User A adds a sentence and User B edits it, all users see the addition before the edit, preserving intention.
#### Chat Applications and Social Media: 
> Ensures replies to a message are seen after the original message. If User A comments on a post and User B replies, everyone sees the comment before the reply.
#### Distributed Databases (e.g., MongoDB, Redis): 
> Used in geo-replicated databases to ensure that write operations that depend on previous ones are applied in order, preventing users from reading stale data after a new update, particularly when using MongoDB's client sessions with causal consistency enabled.
#### Shopping Carts and Inventory: 
> Used to maintain order in related updates, such as ensuring an item is added to a cart before a "checkout" action, ensuring consistent shopping experiences across regions.
#### Configuration Management: 
> Systems that manage changes in configurations where a subsequent setting depends on a previous one.
